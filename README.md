## 📖 О проекте
**Student Alarm Clock** — зачётный проект по дисциплине «Мобильная разработка» (2025). 
Целью было создание удобного, кастомизируемого будильника, закрывающего личные потребности. В процессе разработки был реализован полный цикл работы с локальной БД, сетевыми запросами (погода) и, самое главное, сложными системными API Android для гарантированного срабатывания фоновых задач в условиях агрессивных ограничений современных версий ОС.

## ✨ Ключевые особенности
- **Надежное срабатывание:** Использование `AlarmManager.setAlarmClock()` для обхода Doze Mode на Android 12+.
- **Кастомный UI срабатывания:** Полноэкранная `AlarmActivity` с отображением поверх экрана (`SYSTEM_ALERT_WINDOW`), вибрацией и воспроизведением системного рингтона.
- **Интеграция погоды:** Получение данных по геолокации через OpenWeatherMap с локальным кэшированием в `SharedPreferences` и загрузкой иконок через Glide.
- **Гибкое планирование:** Настройка повторений (один раз, ежедневно, будни/выходные, кастомные дни).
- **Обработка Edge-cases:** Автоматическая проверка и запуск пропущенных будильников при загрузке устройства (`BootReceiver`) и смене времени/часового пояса (`TimeChangeReceiver`).
- **Мультивыбор:** Удобный режим массового удаления будильников с анимированным тулбаром.

## 🚀 Технические вызовы и инженерные решения
1. **Проблема:** На Android 12+ система блокирует установку точных будильников без явного разрешения пользователя.
   **Решение:** Инкапсулировал проверку `canScheduleExactAlarms()` и запрос `SCHEDULE_EXACT_ALARM` в единый `PermissionManager`. UI-слой не знает о системных ограничениях, а только реагирует на состояние `PermissionState`.
2. **Проблема:** Необходимость разбудить устройство и показать UI поверх лоcksreen.
   **Решение:** Настроил `AlarmActivity` с флагами `showWhenLocked` и `turnScreenOn`. Добавил fallback: если запуск Activity блокируется системой, показывается High-Priority Notification с `Full-Screen Intent`.
3. **Проблема:** Реактивное обновление списка без лишних пересозданий.
   **Решение:** Room DAO возвращает `Flow<List<AlarmEntity>>`, который прозрачно маппится в Repository и конвертируется в `LiveData` для ViewModel.

## ⚠️ Известные ограничения и Roadmap (Технический долг)
Проект находится в стадии активного рефакторинга. Как инженер, я вижу следующие точки роста:
- **Дублирование планировщиков:** В коде сосуществуют legacy `AlarmScheduler` (object) и новый `AndroidAlarmScheduler` (реализующий `IAlarmScheduler`). *План:* Полная миграция `BootReceiver` и `TimeChangeReceiver` на новый интерфейс и удаление старого кода.
- **Кастомный звонок vs Системный:** Сейчас звук и вибрация управляются вручную в `AlarmActivity`. *План:* Переход на `Intent.ACTION_SET_ALARM` для делегирования звонка системному приложению. Это снизит нагрузку на батарею и уберет баги, связанные с фоном.
- **Безопасность секретов:** API-ключ OpenWeatherMap сейчас зашит в `build.gradle.kts`. *План:* Вынос в `local.properties` и использование `secrets.properties` плагина.
- **Тестирование:** Покрытие бизнес-логики (например, расчет `nextTriggerTime` в `AlarmTimeCalculator`) равно нулю. *План:* Написание Unit-тестов для чистых функций.

## 📂 Структура проекта
- `data/` — слой данных (Room DAO, Entities, мапперы, Repositories).
- `service/alarm/` — ядро будильника (Receivers, Services, Schedulers, AlarmActivity).
- `ui/` — презентационный слой (Fragments, Adapters, кастомные диалоги).
- `viewmodel/` — логика UI, управление состояниями (Sealed Classes).
- `weather/` — сетевой слой для интеграции с API погоды.
- `utils/` — утилиты (анимации, менеджер разрешений).
